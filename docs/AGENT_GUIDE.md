> **Если ты — AI-агент (Claude / GPT / Cursor / Copilot), которому передали этот репозиторий:** прочитай этот файл целиком ПЕРЕД тем как помогать с кодом. Здесь всё что я (предыдущий AI-агент) узнал болью и ошибками. Сэкономит тебе часы и предотвратит повторение фейлов.

## Контекст проекта

Этот бот построен **AI-агентом** (Claude в Telegram-чате) совместно с пользователем-разработчиком, который хотел подарить рабочий инструмент **другу-владельцу** хостела (Hostel Dos Estaciones, Аликанте). Друг — не программист. Он клонирует это репо и **скорее всего попросит своего AI-агента (тебя) помочь** с поддержкой / деплоем / изменениями.

**Цель проекта:** WhatsApp-бот собирает данные у гостя через диалог → автоматически бронирует номер на hotelgest.com → возвращает ссылку на оплату Monei.

**Статус:** MVP работает end-to-end (проверено май 2026). Полный flow ~30 секунд от первого сообщения до payment URL.

## Технический стек и почему

| Компонент | Выбор | Альтернатива | Почему такой выбор |
|---|---|---|---|
| WhatsApp клиент | **whatsapp-web.js** | Meta Cloud API | Не нужна верификация бизнеса, бесплатно, быстрый старт. Минус — может отвалиться при обновлениях WhatsApp |
| LLM | **GPT-4o-mini** | Claude / GPT-4o | Дёшево (~$0.15/1M), быстро, у владельца кредитов есть. Задача простая (извлечь 6 полей) |
| Browser автоматизация | **Puppeteer** | Playwright / Selenium | Уже идёт с whatsapp-web.js, один Chromium на всё, без отдельного `npx playwright install` |
| Сессии | **In-memory Map** | Redis / DB | MVP, один процесс, простая логика. Минус — теряются при рестарте |
| Process manager | **PM2** | systemd / Docker | Самый простой, авто-перезапуск, удобный мониторинг |

Подробности → [ARCHITECTURE.md](ARCHITECTURE.md)

## КРИТИЧЕСКАЯ информация о hotelgest.com booking flow

Это что мы узнали через несколько часов боли и визуальной отладки. **НЕ повторяй мои ошибки.**

### Грабли #1: Календарь автоматически открыт модалом

При загрузке страницы датапикер **автоматически открывается оверлеем** и блокирует клики по чему угодно за ним. До декабря-2025 не было такого. Если в booking падает на ранних шагах — первое что проверять.

### Грабли #2: Скрытые `dfrom`/`dto` НЕ работают

Раньше я думал что можно установить `input[name="dfrom"].value = '14-05-2026'` и кликнуть Search — фигня. MobiScroll-виджет хранит свой внутренний state, и поиск использует его, а не скрытые inputs. **Конкретный симптом:** модал номера показывает `Nights: 1` хотя ты выбрал 2 ночи → кнопка Add не появляется (Min Stay: 2 не выполнен).

**Правильный путь:** кликать даты через календарь по `[aria-label="Thursday, May 14, 2026"]`, потом кнопку Accept.

### Грабли #3: `waitForSelector` без `{ visible: true }`

Поля формы `#booking-customer-name` и т.д. **существуют в DOM с самого начала загрузки страницы** (просто скрытые). `waitForSelector(...)` без `visible: true` резолвится МГНОВЕННО → ты пытаешься заполнить форму до того как она появилась → submit падает.

**Всегда** используй `{ visible: true }` для интерактивных элементов.

### Грабли #4: Старые `.modalRate` от дефолтной выдачи

Когда страница загружается, она показывает rooms для **дефолтных дат** (сегодня + 1 день). Класс `.modalRate` уже на странице. Когда ты кликаешь Search для НОВЫХ дат, новые rooms перезаписывают старые. Но `waitForSelector('.modalRate', { visible: true })` РЕЗОЛВИТСЯ МГНОВЕННО на старом элементе → ты кликаешь stale element.

**Решение:** после Search → `await page.waitForNetworkIdle({ idleTime: 800 })` + 500ms буфер.

### Грабли #5: Phone — два поля!

`#booking-customer-phone` это только **цифры** без кода страны. Префикс — отдельный `<select id="booking-customer-prefix-phone">` с 248 опциями. По умолчанию `+41` (Швейцария!), нужно сменить на `+34` для Испании.

В `setPhoneFields()` есть логика автоматического разбиения телефона по префиксу.

### Грабли #6: `#booking-customer-country` — НЕ select!

Это **text input** (autocomplete). `page.select('#booking-customer-country', 'ES')` упадёт. Используй `typeField(page, '#booking-customer-country', 'Spain')` — но это поле необязательное, можно пропустить.

### Грабли #7: Submit не редиректит — Monei встроен inline

Когда нажимаешь Submit, страница **не уходит** на Monei. Вместо этого Monei JS SDK внедряет card-форму прямо на текущую страницу. Для бота это не работает — гость на WhatsApp, ему нужен URL.

**Решение (ключевое):**
- Перехватываем сетевой запрос `https://api.monei.com/v1/payment-methods?paymentId=XXX` (Monei JS дёргает этот endpoint при инициализации)
- Извлекаем `paymentId`
- Строим URL `https://secure.monei.com/payments/{paymentId}` — это standalone hosted-страница оплаты, можно отправить в WhatsApp

### Грабли #8: Unhandled promise rejection крашит Node 22

`paymentPromise` может реджектиться по таймауту в любой момент booking flow. Если в этот момент мы `await`-им что-то другое — реджект unhandled → краш процесса. **Всегда** добавляй `paymentPromise.catch(() => {})` сразу после создания.

## Что хорошо проверять перед изменениями

### Перед тем как менять `booking.js`:
1. **Запусти локально с `headless: false`** — посмотри глазами что происходит
2. Включи `DEBUG=1` для скриншотов на каждом шаге
3. Если меняешь селекторы — обнови JSDoc в начале файла
4. Тестируй с РАЗНЫМИ датами (1 ночь / несколько ночей / далекие)

### Перед тем как менять `bot.js`:
1. **Не убирай tool-calling** — это защита от jailbreak. LLM в свободном режиме небезопасен.
2. **Не ослабляй system prompt** — там жёсткие ограничения на off-topic.
3. **Не убирай session limits** — защита от абуза.
4. Если меняешь BOOKING_TOOL — обновляй и обработку args в processMessage.

### Перед деплоем:
1. `npm install` (всегда после `git pull`)
2. Проверь .env (есть ли OPENAI_API_KEY)
3. Если меняли `sessionDir` — заново сканируй QR
4. `pm2 restart hostel-bot && pm2 logs --lines 50` — смотри что стартанул

## Типичные задачи, которые попросит друг

### "Бот перестал отвечать"
1. `pm2 status` — процесс жив?
2. `pm2 logs hostel-bot` — есть ошибки?
3. Проверь WhatsApp на телефоне — сессия активна? (на телефоне в Связанных устройствах должен быть активный)
4. `pm2 restart hostel-bot`

### "Бронирования стабильно падают"
Hotelgest, наверное, обновил сайт. Запускай `headless: false` локально, смотри где падает, обновляй селекторы.

### "Хочу чтобы бот говорил больше / по-другому"
Меняй `SYSTEM_PROMPT` в `bot.js`. **Не убирай ограничения**.

### "Хочу другие лимиты"
Меняй `session.maxMessages` / `timeoutMinutes` в `config.js`.

### "Сменить язык бота на другой основной"
Меняй `bot.language` + `bot.welcomeMessage` + `bot.limitMessage` в `config.js`.

### "Добавить ещё один отель"
Текущая архитектура — один бот = один отель. Для нескольких отелей нужно:
- Множить boot-folder для каждого
- Или делать multi-tenant (несколько urlов, выбор по WhatsApp-инстансу) — серьёзный рефакторинг

### "Хочу email-уведомления когда кто-то бронирует"
Добавь nodemailer в bot.js, после успешного `bookRoom()` шли email админу.

### "Сохранять историю всех бронирований"
Простой вариант: append в JSON-файл / SQLite. Серьёзный — PostgreSQL.

## Чего делать НЕЛЬЗЯ

- ❌ **Не коммить .env** в git (уже в .gitignore, но проверяй)
- ❌ **Не сканируй QR с личного WhatsApp** — нужен ОТДЕЛЬНЫЙ номер для бота
- ❌ **Не убирай anti-abuse** (tool-calling, session limits, system prompt restrictions)
- ❌ **Не делай `npm install` без проверки `package-lock.json`** — может протянуть несовместимую версию whatsapp-web.js
- ❌ **Не запускай `npm install -g`** — всё локально в проекте
- ❌ **Не меняй selectors в booking.js без визуальной проверки** — снова попадёшь в те же грабли что я

## История разработки (контекст)

Этот проект родился из переписки в Telegram между владельцем-программистом и AI-агентом. Параллельно агент помогал владельцу с гардеробом и стиркой одежды (без шуток). Потом владелец вспомнил про друга-хостельера и попросил собрать MVP бота для бронирований.

Ключевые этапы:
1. Архитектура и стек обсудили в чате
2. Выбрали whatsapp-web.js для скорости (vs Meta Cloud API с верификацией бизнеса)
3. Решили GPT-4o-mini вместо Claude (дёшево, и кредиты есть)
4. Tool-calling вместо свободного диалога — защита от абуза
5. Booking automation — взяли puppeteer (уже идёт с whatsapp-web.js)
6. Несколько итераций отладки hotelgest.com (см. "Грабли" выше)
7. End-to-end тест прошёл — бот вернул реальный Monei URL гостю в WhatsApp

Все коммиты в git history — там видно прогресс через ошибки.
